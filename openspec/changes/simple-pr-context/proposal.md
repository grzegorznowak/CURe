# Proposal: simple-pr-context

## Goal / Context

Add an opt-in `cure pr` pilot that orients supported built-in reviews from bounded discussion on the selected pull request. The system fetches and normalizes the selected PR's issue comments, reviews, and inline review comments; preserves that complete ordered remote corpus unchanged as an audit artifact; deterministically selects a bounded model-input subset; produces a bounded orientation brief; and delivers at most 2,000 estimated tokens of `$PRIOR_CONTEXT` to supported review reconciliation.

The CLI exposes `--pr-context` / `--no-pr-context` and defaults off. Unsupported or custom prompt/profile flows bypass before fetch/orientation. Historical `--if-reviewed list|latest|prompt-selected` returns and `--no-review` remain non-review routes even with explicit enablement: they perform no PR-context work and create no PR-context artifact or metadata. PR-context enrichment errors warn visibly and fall back to the ordinary context-free review path without swallowing unrelated review or process-control failures. Completion enables only an opt-in pilot; default-on/general release requires separate operator approval based on context-on/context-off evidence.

## Story Candidates

Single story — this change workspace is the full scope of the `simple-pr-context` opt-in pilot.

## Decisions & Constraints

- **MVP source:** selected-PR remote discussion only. Local CURe session/history discovery and recovery are removed from current scope.
- **Audit/model split:** `work/pr_context_discussion.json` preserves every normalized endpoint event unchanged and in endpoint order. The audit corpus is not itself the orientation input contract.
- **Deterministic bounds:** 12,000 estimated tokens for the fully assembled orientation-generation prompt, including fixed instructions, canonical framing, normalized PR stats, and serialized selected normalized event records; 1,000 estimated tokens per selected event body; 100 selected events; 2,000 estimated tokens orientation output; 2,000 estimated tokens injected `$PRIOR_CONTEXT`. Estimates use `ceil(len(text) / 4)` and are labeled estimates.
- **Selection:** consider timezone-aware parse-valid timestamps newest-first; ties use endpoint ordinal then source index ascending, while missing/empty/naive/invalid timestamps follow valid timestamps in endpoint/source order. Prefix-cap each candidate body, admit only within the budget remaining after fixed prompt/framing overhead, stop at the first full-prompt or count limit, then restore valid timestamps oldest-first with the same tie-breakers followed by invalid/missing events in endpoint/source order. Compact deterministic JSON and fixed separators define exact cap-1/cap/cap+1 assembly. Report selected, omitted, and body-truncated events and token estimates.
- **Supported delivery:** built-in singlepass uses its ordinary blind draft followed by context reconciliation; built-in multipass keeps plan/step context-free and supplies context only to synthesis. Both regular and completed-session incremental multipass resume routes reuse `work/pr_context_orientation.md` only when originating session `meta.json::pr_context.outcome == "used"` and the exact brief is valid, structurally finalized, and within the 2,000-token injection cap. Missing, bypassed, degraded, legacy/no-outcome, invalid, or over-cap persisted state resumes context-free without discussion fetch or orientation. A completed/same-head fast no-op or reusable-review-artifact resume that performs no synthesis/reconciliation preserves the originating `pr_context` value without key/value mutation rather than creating a new context-on/off record.
- **Fail-open boundary:** fetch, normalization/selection, orientation, required pre-delivery discussion/brief persistence, context insertion/reconciliation, and context-specific synthesis are enrichment boundaries. Their errors warn and produce `degraded` metadata while ordinary review continues. A singlepass reconciliation failure retains the blind draft. A context-specific multipass synthesis failure in fresh, regular-resume, or incremental-resume delivery retries once with empty context from the same successful plan/steps; successful fallback records `degraded/context_synthesis_failed`, `context_mode=off`, combined delivery latency, and separate nullable delivery/fallback provider-usage fields. Failure of that empty-context retry propagates while authoritative session metadata retains available degradation telemetry. Final `work/pr_context_meta.json` persistence happens once after a successful route completion (delivery, successful fallback, or bypass) has final telemetry; its failure warns and is retained in authoritative session metadata without retrying review or replacing the route outcome/reason. Core session metadata flush and failures in ordinary fallback, checkout, prompt/profile validation, draft/plan/step execution, output acceptance/posting, cancellation, or other process control are not swallowed.
- **Eligibility:** the feature is evaluated before fetch. Explicitly disabled, unsupported profile, custom prompt, or custom prompt-file flows record `bypassed` and make no discussion/orientation call. Historical `--if-reviewed list|latest|prompt-selected` exits return before session creation; `--no-review` preserves its session/index-only completion. Both are outside the invocation boundary and create no context call/artifact/metadata. Zero normalized remote events records `bypassed/no_remote_context`; a nonempty normalized audit corpus from which the bounded selector admits zero events records `bypassed/no_selected_context`, preserves the complete audit artifact, and skips orientation.
- **Observability:** every route that enters a PR-context delivery decision records stable metadata with `outcome=used|bypassed|degraded`, deterministic path/reason precedence, enabled/eligible state, `enablement_source=default|cli_explicit`, counts, estimated tokens, provider usage when available, truncation, persistence state, stage latency, and context mode. Fresh fetched/normalized counts have separate completed-boundary definitions. Resume metadata inherits sanitized acquisition provenance only from an originating `used` run, resets it for every other origin, and records delivery/result/mirror fields from the current attempt. Session `meta.json::pr_context` is authoritative; `work/pr_context_meta.json` is a best-effort final mirror. Resume exits that perform no delivery preserve that authoritative origin value deep-equal with no key/value mutation and make no new mirror attempt. Estimates and provider usage are separate fields.
- **Pilot comparison:** operators can run otherwise comparable context-on/context-off reviews and compare run-level review output, duplicate/context-invalid comment incidence, cost, latency, and degradation. No finding identity/disposition model is introduced.
- **Release gate:** this story authorizes only opt-in pilot use. Default-on/general release is a separate operator-approved change based on evidence; this plan invents no arbitrary quality threshold.

## Explicit Non-goals

Finding identity/disposition, authoritative resolution, local history, freshness/caching, richer reconciliation analytics, persistent review memory, and general-release enablement.
