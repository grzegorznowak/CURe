# Proposal: simple-pr-context

## Goal / Context

`cure pr` currently reviews PRs blind to discussion history — it sees only the PR description and the diff. This change adds a `cure_pr_context` package that fetches all PR discussion (comments, reviews, review comments) and past CURe reviews, deduplicates them, and runs a single LLM orientation scan. The resulting structured brief is delivered according to the actual review topology: every fresh or resumed render of the shared multipass synth template receives `$PRIOR_CONTEXT` (current/persisted brief or `""`); singlepass with context uses two real LLM calls (draft review without context, then `_reconcile_prior_context(draft_review, orientation_brief, run_llm)`); singlepass without context remains the current one-call path. The feature runs automatically on `cure pr` review runs with no operator flags required.

## Story Candidates

Single story — this change workspace is the full scope of the `simple-pr-context` initiative.

## Decisions & Constraints

A single `cure_pr_context/` package with 4 files plus a small top-level `cure_github.py` adapter module. The public API is `build_pr_context(pr, sandbox_root, work_dir, pr_stats, head_sha, gh_fetch, run_llm)`, which returns a dict with the orientation brief, deduplicated debug data, and metadata. `head_sha` is the effective PR/review SHA passed explicitly from `_pr_flow_impl` to annotate prior-review head metadata (`reviewed_head`, `current_head`, and match status), never to exclude same-PR reviews from earlier heads; `gh_fetch` is list-capable and based on `cure_github.gh_api_list` (not `gh_api_json`), `sandbox_root` is the real sessions/sandboxes root, `work_dir` is the current session's `work/`, and `pr_stats` is the result already computed by `compute_pr_stats`. Remote footer/session metadata that indicates a different PR is still ignored. The brief is produced by a single LLM scan with fixed sections and inline usage instructions. `$PRIOR_CONTEXT` appears only in the multipass synth template; normal/big singlepass, multipass plan, and multipass step templates have no token. Singlepass with non-empty context runs the ordinary singlepass prompt first, then `_reconcile_prior_context(draft_review, orientation_brief, run_llm)` performs the second LLM call and returns the final review. Singlepass with `orientation_brief == ""` remains one call. Multipass plan and step calls remain independent. Fresh synth uses the current orientation brief; interrupted synth resume reuses persisted `work/pr_context_orientation.md` when present and supplies `""` when absent, so the shared template never leaks a raw token. The three REST discussion endpoints do not expose authoritative review-thread resolution state; `Resolved areas` summarizes only areas that discussion or past-review content describes as addressed or resolved. Past-review deduplication collapses identical local/posted representations, then uses character 3-grams + Jaccard ≥ 0.85 (inclusive) with no external dependencies; `past_reviews` is the retained side and duplicate events are removed from the returned/written/LLM-passed `discussion`. The LLM is received as a `Callable` injected from `cure.py`. Any error aborts the review. No cache, no new CLI flags, no changes to `cure_flows.py`.
