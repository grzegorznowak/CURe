## Purpose

Connect subsequent-review intake and semantic ledgers to CURe's runtime review prompts, pre-review intelligence, and operator-facing degraded-mode UX. A subsequent-aware `cure pr` run should: inject the governor's pre-review intelligence brief into the LLM prompt so expensive re-analysis of already-discovered findings is avoided; reuse decision-phase discussion evidence through a single fetch to prevent fetch divergence (FB-010); persist accumulated finding memory per PR so future runs skip redundant LLM verification of unchanged confirmed findings; persist source-verification cache state as soon as intake/source verification succeeds so failed or still-running reviews do not lose expensive verifier work; replay safe terminal non-reportable prior outcomes when their identity/head/provenance is unchanged; present operator-controlled retry/skip/abort handling before any degraded PR-discussion artifacts are committed; perform a post-review sanitization LLM call to audit coverage of prior review context; require final `review.md` to explain prior-review state as human-readable issue history rather than raw DA-number lists; and prove the complete local simulation trace through deterministic fixtures. The pre-review brief and memory are intelligence inputs, not mechanically-injected output — the final review remains the LLM's own analysis. The packager (`review_context_package.json`, `subsequent_review_context.md`) serves audit, debug, and sanitization purposes but is NOT injected into the LLM prompt.

Findings where `last_seen_head == current_head AND source_state == "resolved_from_source"` are partitioned out and synthesized from memory (no LLM call); findings with safe terminal non-reportable/disposition-only cache entries may also be partitioned when current head, stable identity, source-reference identity, and relevant discussion/disposition fingerprints all match. Remaining findings go through the LLM verifier. The final result merges cached + freshly-verified rows. Synthesized rows carry `provenance: "memory_cache"` / `not_source_proof: true` unless replaying a source-resolved row; these are audit-only labels with no functional influence downstream. Aggregation rule: a group is skipped only if ALL linked discussion events carry a skip-class signal with untrusted authority. Any event with `DEVELOPER_CLAIM_FIXED`, `RESOLVED_THREAD_HINT`, `UNRESOLVED_THREAD_HINT`, `AUTHORITY_CONFLICT`, or any TRUSTED authority → verify.

The sanitization call is skipped when the brief is empty (no prior findings).

- A13: Verifier filter from linker signals: reconciliation groups where ALL linked discussion events carry `PUSHBACK`, `BY_DESIGN`, `ADDRESSED_ELSEWHERE`, or `DUPLICATE_SUPERSEDED` with UNTRUSTED authority are skipped (no verifier LLM call). Any event with `DEVELOPER_CLAIM_FIXED`, `RESOLVED_THREAD_HINT`, `UNRESOLVED_THREAD_HINT`, `AUTHORITY_CONFLICT`, or any TRUSTED authority → group is verified. The filter is adjustable (freely tunable signal-class / authority matrix in code).
- A14: LLM verifier is a production `FindingVerifier` in `cure_subsequent_review/llm_verifier.py`. Input per group: Variant C (finding body + PR files-changed + discussion signals + current source from hybrid read: direct file ±20 lines with chunkhound fallback). Two-tier: first LLM pass returns `resolved_from_source`/`still_open`/`need_more_context`; `need_more_context` triggers chunkhound research followed by a second LLM pass returning `resolved_from_source`/`still_open` (never `source_unknown` from second pass). `not_verifiable` only when evidence references don't exist in the repo. Chunkhound tool available directly to the verifier LLM. Per-group verifier results cached in `cure_memory.json` by stable finding identity/fingerprint/source-reference identity plus current head, with ordinal `group_id` retained only as provenance/replay metadata.
- A15: Runtime package/context status consistency: after packager, governor, and memory-store runtime updates, `review_context_package.json`, `subsequent_review_context.md`, `run_manifest.json`, and `meta.json` must agree on final module statuses. Package/context output is refreshed or otherwise written after runtime status mutations so it cannot report successful modules as `disabled` from an early manifest snapshot.
- A16: Final subsequent-review prompts/reports require an explicit human-readable prior-review issue summary. The main `review.md` must group `disposition_ledger.json` DA records into stable issue clusters with canonical titles, current status, and plain-English reasons; leading `### Prior Review Issue History` bullets must preserve the governor-supplied reason, not only title/status. Raw `DA-*` IDs are internal provenance anchors only: they may appear in optional details, machine-readable metadata, comments, or an appendix, but not as the primary frontend representation. The underlying governor/audit coverage must still account for every `DA-*` row with one of: `confirmed-resolved`, `carried-forward/re_report`, `degraded`, `out-of-scope`, or `contradicted-with-evidence`.
- A17: Report-governor post-review awareness checks detect missing/partial human issue-cluster awareness, missing required issue-history reasons, omitted foreign-footer audit notes from final output, missing internal `DA-*` coverage, and contradictions, then record `status="degraded"`, warnings, or explicit partial-awareness evidence. They never fail or block publication in `strict`, `warn`, or `off`; strict mode only strengthens detection and pre-prompt ledger requirements.
- A18: Source verifier/source-truth checks avoid stale duplicate symbol confusion by resolving active import/re-export bindings where possible, or degrading when source references point at inactive/dead duplicate definitions rather than treating stale code as still-open evidence.
- A19: Runtime context/governor carries Story 02 / FB-026 policy provenance: for remote PR comments and pull-review events, an official CURe footer is necessary but not sufficient to identify usable prior CURe review provenance. After footer authenticity is recognized, the event is accepted only when its footer/session/PR/head provenance and any GitHub pull-review event `reviewed_head`/`commit_id` provenance are each compatible with the current review target/run; generic/body-only CURe-looking text remains rejected; and foreign official-footer events (cross-PR, cross-session, mismatched footer SHA, or mismatched pull-review event head) are excluded from the prior corpus and downstream prior-finding/source-verification/disposition ledgers with visible ignored-comment audit reasons. When `footer_marker_policy.foreign_footer_ignored_comments > 0`, package/context/governor/final-review surfaces must include the ignored count and plain-English reason; the final review must include a concise reader-facing provenance/audit note while excluding the foreign findings. Final reviews and the semantic/disposition/governor ledgers must not carry forward intended official-footer acceptance as spoofing unless there is evidence that body-only/generic markers are admitted.
- A20: Source-verification cache hardening prevents repeated PR #22-style verifier storms. Successful enabled intake persists source-verification memory immediately after `source_verification.json` / `disposition_ledger.json` are written, before final `review.md` or post-review governor publication. Cache replay uses stable finding identity/fingerprint/source-reference identity plus current head as the primary key; ordinal `group_id` alone is never sufficient. Replay covers same-head `resolved_from_source` rows and safe terminal non-reportable outcomes such as duplicate/out-of-scope/dropped-not-relevant dispositions when their stable identity, source-reference, discussion/disposition fingerprints, and policy provenance still match. Non-source-proof replay must carry `not_source_proof: true` and must not emit `resolved_from_source` unless the cached source row was source-resolved. Reportable still-open outcomes, source-unknown rows, and not-verifiable rows are not safe terminal replay categories; they must miss/bypass and run fresh verification rather than converting discussion/disposition cache data into source proof. Every group records cache `hit|miss|bypass` with reason codes in source-verification/review-memory artifacts, and large verifier fan-out is observable through manifest/log timing so future runs can prove cache effectiveness or trigger bounded parallelism/cheaper verifier policy work.
- A21: Operator-facing rejected-remote-marker CTA: when decision/control-plane processing observes official-footer remote CURe markers, records `remote_cure_markers > 0`, rejects every marker so `accepted_remote_cure_markers == 0` for current-run provenance mismatch (foreign PR/session/head/footer SHA or pull-review event `reviewed_head`/`commit_id` mismatch), and subsequent review remains disabled, CURe must still surface an immediately actionable operator-only notice before any review body is displayed or copied. Body-only/generic CURe-looking text without an official footer is out of scope for this MVP: it must remain a non-positive signal (`remote_cure_markers == 0`), must not enable subsequent-review intake, and would require a future separate non-positive audit signal if an operator notice is desired. The notice must be outside `review.md`, outside prompt variables, and outside the copyable GitHub review block; must start with an explicit frame such as `CURe Operator Notice — Not part of the review`; and must be emitted by `_pr_flow_impl` handing durable disabled-path rejection metadata to the existing stderr/TUI output renderer/formatter immediately before rendered review markdown, without changing success stdout. Each rejected marker must be persisted as sanitized metadata in `work/subsequent/decision.json.rejected_remote_cure_markers[]` and mirrored in `meta.json.subsequent_review.rejected_remote_cure_markers[]`, with source type, comment/review id, optional URL, author, created/submitted timestamp, reason, current PR/head, footer PR/session/reviewed head, and event reviewed head where available. Rejected body text must never be persisted into that metadata, injected into prompts, or copied into final review output; rejected markers never create runtime/intake artifacts or enable intake by themselves.
