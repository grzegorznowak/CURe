## Purpose

Connect subsequent-review intake and semantic ledgers to CURe's runtime review prompts, pre-review intelligence, and operator-facing degraded-mode UX. A subsequent-aware `cure pr` run should: inject the governor's pre-review intelligence brief into the LLM prompt so expensive re-analysis of already-discovered findings is avoided; reuse decision-phase discussion evidence through a single fetch to prevent fetch divergence (FB-010); persist accumulated finding memory per PR so future runs skip redundant LLM verification of unchanged confirmed findings; present operator-controlled retry/skip/abort handling before any degraded PR-discussion artifacts are committed; perform a post-review sanitization LLM call to audit coverage of prior review context; require final `review.md` to explain prior-review state as human-readable issue history rather than raw DA-number lists; and prove the complete local simulation trace through deterministic fixtures. The pre-review brief and memory are intelligence inputs, not mechanically-injected output — the final review remains the LLM's own analysis. The packager (`review_context_package.json`, `subsequent_review_context.md`) serves audit, debug, and sanitization purposes but is NOT injected into the LLM prompt.

## Triggering Need

Stories 01-03 gather prior review material, infer subsequent-review mode, reconcile prior findings, verify source truth, resolve discussion signals, and arbitrate semantic dispositions. The remaining epic gap is runtime integration: final review prompts currently have no path to surface those ledgers as intelligence input; decision-phase and intake-phase discussion evidence can diverge (FB-010); every new run re-runs expensive LLM source verification for findings that haven't changed; degraded API interactions lack interactive operator control; and the local simulation trace has no deterministic coverage.

- Inject the governor's pre-review intelligence brief into all subsequent-aware PR review prompt paths via one new prompt variable: `$PRIOR_REVIEW_BRIEF`. The variable is injected using the existing `extra_vars` dict pattern (same as `$REVIEW_INTELLIGENCE_GUIDANCE`) at all 5 `render_prompt()` call sites in `_pr_flow_impl`. When subsequent review is disabled or governor mode is `off`, it renders as an empty string. - Pipeline gate owner: `cure_subsequent_review/semantic_pipeline.py` filters the finding list before `verify_source_truth()`. Findings where `last_seen_head == current_head AND source_state == "resolved_from_source"` are partitioned out and synthesized from memory (no LLM call); remaining findings go through the LLM verifier. The final result merges cached + freshly-verified rows. Synthesized rows carry `provenance: "memory_cache"` — an audit-only label with no functional influence downstream. The public API adds `memory_store: ReviewMemoryStore | None = None` and `current_head: str | None = None`; when memory is absent, `current_head` is `None` or empty/blank, or any error occurs, the gate is off and all findings go through full verification. The gate condition explicitly guards against empty/malformed head SHAs.
- Pipeline reorder: `resolve_discussion_signals` now runs BEFORE `verify_source_truth` in the semantic pipeline (was after). This enables the verifier to skip groups that discussion signals confirm as `PUSHBACK`, `BY_DESIGN`, `ADDRESSED_ELSEWHERE`, or `DUPLICATE_SUPERSEDED` with untrusted authority. Aggregation rule: a group is skipped only if ALL linked discussion events carry a skip-class signal with untrusted authority. Any event with `DEVELOPER_CLAIM_FIXED`, `RESOLVED_THREAD_HINT`, `UNRESOLVED_THREAD_HINT`, `AUTHORITY_CONFLICT`, or any TRUSTED authority → verify.
- Implement LLM-based discussion linker: replaces the existing keyword-based `_classify_signal()` with an LLM call per discussion event. Each call receives the event (author, body, thread_state) + prior findings with full bodies. Returns `group_ids` + `signal_class` + `rationale`. Topical matches without confident finding IDs return `group_id: null`. Linker results are cached in `cure_memory.json` by `event_id + body_hash + head`; unchanged events at the same HEAD replay cached results without LLM calls. The existing `DiscussionLinker` injection point in `resolve_discussion_signals()` already supports this — wire it as the production linker.
- Interactive API failure handling: `_pr_flow_impl` constructs a module-12 `DiscussionFetchController` that wraps `collect_pr_discussion()`. The controller fetches once; after the fetch completes but before any artifact is committed, if the result is degraded it presents `[R]etry / [S]kip / [A]bort`. TUI: unlimited retries. Non-interactive: max 3 retries. The controller's accepted/skipped `DiscussionArtifact` is then passed into `decide_subsequent_review(..., discussion=artifact)` (which returns a tuple `(SubsequentReviewDecision, DiscussionArtifact | None)` without writing `decision.json` internally). `_pr_flow_impl` conditionally writes `decision.json` only after the operator gate passes. The same artifact is then passed to `run_subsequent_review_intake(prefetched_discussion=artifact)` so intake never re-fetches. Skip records `operator_skipped_degraded_discussion`; abort exits before prompt with no downstream artifacts.
- Address FB-007 via the pre-review brief: all dispositioned findings are surfaced with provenance. Post-review coverage is audited by a sanitization LLM call (same model as main review) that answers: "Does this review demonstrate awareness of the prior review context?" — producing a qualitative judgment with evidence in `report_governor_result.json`. The sanitization call is skipped when the brief is empty (no prior findings).
- Final-output humanization: final subsequent-review `review.md` must present prior-review awareness as human-readable issue history first. The main report groups internal `DA-*` rows into stable issue clusters with titles, statuses, and plain-English reasons; raw DA IDs remain internal/provenance anchors and must not be the primary reader-facing representation. The latest live audit showed that top-level `### Internal DA coverage` is too prominent for ordinary consumers; Story 04 owns FB-030 by requiring DA coverage to be removed from the ordinary visible review body or demoted to a clearly audit-only surface while preserving complete governor/audit coverage.
- Address FB-010: `decide_subsequent_review()` returns `tuple[SubsequentReviewDecision, DiscussionArtifact | None]`. `_pr_flow_impl` passes the discussion artifact through to `run_subsequent_review_intake(prefetched_discussion=artifact)`. Intake uses it directly instead of calling `collect_pr_discussion()`. Single API call, shared event data in `decision.json` and `pr_discussion.json`.
