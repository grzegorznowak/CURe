## Technical Decisions
- Story 04 is the final runtime integration story for epic modules 9-12. Module 13 is test-only.
- A new `cure_subsequent_review/runtime.py` orchestrates modules 9-12 with explicit `pre_prompt()` and `post_review()` phases; they do NOT extend `semantic_pipeline.py`'s `MODULE_REGISTRY`.
- One new prompt variable: `$PRIOR_REVIEW_BRIEF` (governor, intelligence). Renders empty when subsequent review is disabled or governor mode is `off`. Injected via existing `extra_vars` dict pattern at all 5 `render_prompt()` call sites in `_pr_flow_impl`.
- The Review Context Packager (module 9) produces `review_context_package.json` and `subsequent_review_context.md` for audit, debug, and sanitization-LLM consumption. It reports `present|degraded|missing` per upstream artifact and never fails. It is NOT injected into the review LLM prompt.
- The Report Governor produces a pre-review intelligence brief — it is NOT mechanically injected into the final report. The pre-review pass in `strict` mode raises `ReviewflowError` on missing required inputs before the LLM is called. The post-review pass is a sanitization LLM call (same model as main review) that audits prior-review coverage in the final `review.md`; it is always warn-only (never blocks) and skipped when the brief is empty.
- Human-first final output: DA IDs are internal provenance anchors, not reader-facing labels. Final `review.md` must summarize prior-review state as issue clusters with canonical titles, statuses, and reasons; raw DA-number sets may appear only in optional details, machine-readable metadata, comments, or an internal appendix. Latest live-audit decision: a prominent top-level `### Internal DA coverage` section is too consumer-facing; Story 05 owns demoting/removing that section from ordinary review body while preserving audit coverage.
- Reopened PR #22 follow-up decision: report-governor final-output awareness gaps degrade/warn only and never block publication, even in `strict`; Story 04 owns the propagation of Story 02/FB-026 footer-marker policy into package/governor/final-output checks and into semantic/disposition/governor classification so policy-approved official-footer findings are not carried forward as `re_report`.
- `cure_memory.json` lives at a shared per-PR path `~/.local/state/cure/pr/<host>/<owner>/<repo>/<number>/cure_memory.json`, not per-sandbox. Writes are atomic (temp-file + rename); concurrent writes are best-effort (no locking).
- Pipeline gate ownership: `semantic_pipeline.py` owns the skip before `verify_source_truth()` via `run_semantic_pipeline(..., memory_store: ReviewMemoryStore | None = None, current_head: str | None = None)`. It skips LLM verification only for findings where `last_seen_head == current_head AND source_state == "resolved_from_source"`; the gate defaults to off.
- FB-010: `decide_subsequent_review()` returns `tuple[SubsequentReviewDecision, DiscussionArtifact | None]` and does NOT write `decision.json` internally. `_pr_flow_impl` destructures the tuple, conditionally writes `decision.json` after the operator gate, and passes the discussion artifact to `run_subsequent_review_intake(prefetched_discussion=artifact)`. Intake uses the prefetched artifact directly instead of calling `collect_pr_discussion()`. `collect_pr_discussion()` is called exactly once per enabled run.
- Degraded Runtime Manager: `_pr_flow_impl` builds a `DiscussionFetchController` that wraps `collect_pr_discussion()`. The controller fetches once, presents the degraded dialog BEFORE any artifacts are committed, and passes the accepted/skipped `DiscussionArtifact` into `decide_subsequent_review(discussion=artifact)` and `run_subsequent_review_intake(prefetched_discussion=artifact)`. Neither function calls raw `collect_pr_discussion()` when an artifact/fetcher is supplied.
- API degradation (`discussion_unavailable`, `discussion_incomplete`, `discussion_payload_malformed`) triggers interactive retry/skip/abort dialog at the controller fetch boundary — before any `decision.json`/`pr_discussion.json` commit. TUI: unlimited retries. Non-interactive: max 3 retries. Retry discards the tentative result and re-fetches. Skip records `operator_skipped_degraded_discussion` in `decision.json`/`degraded_runtime.json`; creates no `pr_discussion.json`/runtime artifacts/prompt vars when no other prior signal exists; writes synthetic no-history `pr_discussion.json` only when completed-session evidence enabled intake. Abort records `degraded_runtime.json` and exits before prompt execution.
- Governor mode (`strict`, `warn`, `off`) is configured via new TOML key `[subsequent_review].governor_mode`, parsed by `cure.py`'s `load_reviewflow_subsequent_review_config()` (or equivalent), defaulting to `"strict"`, and rejecting invalid/non-string values with `ReviewflowError`. `_pr_flow_impl` passes the parsed mode into runtime hooks. `strict` affects pre-review brief completeness before prompt rendering, `warn` degrades/continues on incomplete brief inputs, and `off` skips module 10 and renders `$PRIOR_REVIEW_BRIEF` empty. The post-review pass, when active, is always warn-only.
- Disabled subsequent-review path preserves Story 02's decision-only contract: no runtime `run_manifest.json`, packager, governor, memory, or degraded artifacts are created; `work/subsequent/decision.json` and `meta.json.subsequent_review` explain auto-disabled or explicit-disabled state.
- Evidence policy remains exactly `trusted` and `untrusted`.
- Arbiter actions remain exactly the existing five. Degraded findings remain separate.
- PR HEAD SHA is validated at extraction (`cure.py` L9597-9599): blank or missing `head_sha` raises `ReviewflowError`. This ensures `current_head` is always a valid hex SHA for the memory gate. Empty/blank SHAs are additionally guarded at the memory gate level.
- Memory gate `provenance: "memory_cache"` is an audit-only label with no functional influence on downstream consumers (arbiter, governor, prompt builder).
- Sanitization LLM (post-review) uses the same model as the main review for simplicity. Skipped when `governor_brief.md` is empty.
- LLM discussion linker replaces keyword-based `_classify_signal()`; runs per event, parallelizable, cached in `cure_memory.json` by `event_id + body_hash + head`. Wired through the existing `DiscussionLinker` injection point.
- LLM finding verifier (`llm_verifier.py`) is a production `FindingVerifier` with Variant C input, hybrid source reading, two-tier LLM (direct → chunkhound research → final), chunkhound tool available to the LLM. Cached in `cure_memory.json` by `group_id + head`. Replaces `default_finding_verifier` no-op.
- Semantic pipeline reordered: `DISCUSSION_SIGNAL_RESOLVER` runs before `SOURCE_TRUTH_VERIFIER`. Verifier filter skips groups where ALL linked events carry skip-class signals (`PUSHBACK`, `BY_DESIGN`, `ADDRESSED_ELSEWHERE`, `DUPLICATE_SUPERSEDED`) with UNTRUSTED authority.
- Four LLM call sites in Story 04: (1) discussion linker, (2) finding verifier, (3) main review (existing), (4) sanitization (new). Linker and verifier calls are progressively reduced by their respective memory gates on subsequent runs at the same HEAD.
- Live PR #21 is optional/manual only; deterministic local fixtures/goldens own routine verification.

## Implementation Strategy
Recommended red-first sequence:

1. Add contract tests for modules 9-13, artifact schemas, shared memory path derivation, governor subheading enums, and degraded dialog decision types.
2. Add `load_reviewflow_subsequent_review_config()` (or equivalent) in `cure.py` with config-runtime tests for missing/valid/invalid `[subsequent_review].governor_mode`; harden `head_sha` validation at extraction (L9597-9599). Wire metadata/export before module 10 implementation so the mode is not advisory.
3. Implement Review Context Packager with tests over fixture Story 01-03 ledgers, reporting `present|degraded|missing` per upstream artifact, FB-010 evidence reuse (single-fetch marker), and disabled-path empty output. Packager output is audit-only — NOT injected into LLM prompt.
4. Implement Report Governor pre-review brief generation with exhaustive row-ID citation, severity/authority, empty-ledger edge cases (renders as empty string), and strict/warn/off mode behavior (`strict` raises `ReviewflowError` on missing inputs before LLM).
5. Add prompt-render tests proving `$PRIOR_REVIEW_BRIEF` is injected via `extra_vars` in all 11 subsequent-aware PR prompt templates when enabled, empty when disabled/governor-off, and absent from `mrereview_zip.md`.
6. Implement LLM-based discussion linker (`discussion_linker.py`): per-event LLM calls, Variant C-like input (event + prior findings with full bodies), JSON output. Wire through `DiscussionLinker` injection point. Cache linker results in `cure_memory.json` by `event_id + body_hash + head`.
7. Implement LLM-based finding verifier (`llm_verifier.py`): production `FindingVerifier`, Variant C input, hybrid source reading (direct file ±20 lines + chunkhound), two-tier LLM (first pass → resolved/still_open/need_more_context → chunkhound research → second pass → resolved/still_open; `not_verifiable` only when evidence refs don't exist). Chunkhound tool available to verifier LLM.
8. Reorder semantic pipeline: `DISCUSSION_SIGNAL_RESOLVER` before `SOURCE_TRUTH_VERIFIER`. Implement verifier filter: aggregate linker signals per group, skip groups with unanimous skip-class + untrusted authority, verify all others.
9. Implement Review Memory Store: shared path resolution, `cure_memory.json` read/write/update (atomic via temp+rename), two sections (`findings` + `linker_results`), pipeline gate synthesis for cached resolved findings and cached linker signals (partitioning + merge). Gate defaults to off (no memory / empty/blank current head → full verification).
10. Implement Report Governor post-review sanitization LLM call: same model as main review, consumes `governor_brief.md` + `review.md`, produces qualitative judgment + evidence in `report_governor_result.json`. Skipped when brief empty. Always warn-only.
11. Implement Degraded Runtime Manager `DiscussionFetchController`: wraps `collect_pr_discussion()`, fetches once, presents dialog before any artifacts committed, records choices in `degraded_runtime.json`. Decouple TTY detection and fetcher injection for testability.
12. Wire `runtime.py` orchestrator calling modules 9-12 pre/post review. In `_pr_flow_impl`: construct `DiscussionFetchController` before decision/intake, destructure `decide_subsequent_review()` tuple return, conditionally write `decision.json` after operator gate, pass discussion artifact to intake via `prefetched_discussion`, construct and wire LLM linker and verifier, pass parsed governor mode to runtime hooks, pass memory store/current head to semantic pipeline, register `$PRIOR_REVIEW_BRIEF` via `extra_vars` pattern. Preserve Story 02 decision-only disabled behavior.
13. Implement Landmark Trace Runner (test-only) over simulation fixtures/goldens. Module 13 enum entry exists but no runtime registration.
14. Run focused new suites, config-runtime tests, public subsequent-review wrapper, Story 01-03 regression suites (with memory gate default-off), `ruff`, `mypy`.

Follow-up red-first sequence for reopened PR #22 findings F1-F5 / A15-A19:

15. Also add a red semantic/disposition/governor regression equivalent to live `G-0006`/`SV-0006`/`DA-0006`: a prior finding that objects to official-footer acceptance must not remain `still_open` + `re_report` / `carried-forward` when FB-026 says that acceptance is intended. The `semantic_pipeline.py` memory gate must default to off: when no `cure_memory.json` exists, `current_head` is unavailable, or memory is unreadable, all findings go through the LLM verifier normally. Memory is an optimization, not a required dependency.

Implementation should stop and return to planning if the pre-review governor cannot produce a structured brief with deterministic row-ID citations from the disposition ledger, or if the memory gate cannot distinguish matching-head from stale entries without requiring live GitHub data.

## Risks & Mitigations
- Product repo currently has no `AGENTS.md` or `CLAUDE.md`; workspace-level AGENTS instructions apply.
- No epic `CONTRACT.md` exists for `cure-subsequent-pr-review`.
- `projects/CURe/cure_subsequent_review/contracts.py:57-70` already enumerates modules 9-13 in `SubsequentReviewModule` enum.
- `projects/CURe/cure_subsequent_review/control_plane.py:37-46` enables Story 01 modules plus current `MODULE_REGISTRY` entries; modules 9-12 need new `runtime.py` orchestration, not registry extension.
- `projects/CURe/cure_subsequent_review/semantic_pipeline.py:31-47` currently registers only modules 6-8 in MODULE_REGISTRY with `SOURCE_TRUTH_VERIFIER` before `DISCUSSION_SIGNAL_RESOLVER`. Story 04 reorders them: `DISCUSSION_SIGNAL_RESOLVER` first, `SOURCE_TRUTH_VERIFIER` second. Module 13 is test-only, modules 9-12 are in `runtime.py`.
- `projects/CURe/cure_subsequent_review/source_truth.py:16-25` `FindingVerificationRequest` currently has no `pr_files_changed`, `discussion_signals`, or sandbox/repo path fields. The LLM verifier (`llm_verifier.py`) needs these for Variant C input and hybrid source reading. Either extend the protocol with new optional fields or have the verifier access these via its own resources/injected dependencies. Decision deferred to implementation (protocol extension preferred for testability).
- `projects/CURe/cure_subsequent_review/discussion_signals.py:91-96` `resolve_discussion_signals()` currently does not receive a `current_head` parameter. The linker caching gate uses `head` in its lookup key (`event_id + body_hash + head`). The semantic pipeline must pass `current_head` to `resolve_discussion_signals()` or the linker must resolve it independently.
- `projects/CURe/cure_subsequent_review/decision.py:87-190` — `decide_subsequent_review()` already calls `collect_pr_discussion()` for remote probing. Story 04 must place module-12 degraded fetch control around this call before `decision.json` is written, and its accepted/skip `DiscussionArtifact` result must be passed to `run_subsequent_review_intake()` via the new `prefetched_discussion` parameter.
- `projects/CURe/cure.py:9597-9599` extracts `head_sha` from the GitHub API response with a defensive fallback to `""`. Story 04 must harden this: validate the extracted SHA and raise `ReviewflowError` if blank/missing.
- `projects/CURe/cure.py:108-124` owns TOML loading via `load_toml()`, and existing `load_reviewflow_*_config()` helpers near `projects/CURe/cure.py:933-1324` establish the config-owner pattern. Story 04's `[subsequent_review].governor_mode` parser belongs in `cure.py`, should default to `strict`, reject invalid/non-string values with `ReviewflowError`, and be covered by `tests/_reviewflow_unittest_config_runtime_impl.py`.
- `projects/CURe/cure.py:2013-2029` builds review-intelligence guidance; `projects/CURe/cure.py:4268` exposes prompt vars. A new var `$PRIOR_REVIEW_BRIEF` joins `$REVIEW_INTELLIGENCE_GUIDANCE` via the existing `extra_vars` dict pattern.
- Prompt templates already consume `$REVIEW_INTELLIGENCE_GUIDANCE`; add `$PRIOR_REVIEW_BRIEF` after `$REVIEW_INTELLIGENCE_GUIDANCE` in the eleven subsequent-aware PR templates: `default.md`, `mrereview_gh_local.md`, `mrereview_gh_local_big.md`, `mrereview_gh_local_followup.md`, `mrereview_gh_local_big_followup.md`, `mrereview_gh_local_big_plan.md`, `mrereview_gh_local_big_step.md`, `mrereview_gh_local_big_synth.md`, `mrereview_gh_local_big_resume_plan.md`, `mrereview_gh_local_big_resume_step.md`, and `mrereview_gh_local_big_resume_synth.md`. `mrereview_zip.md` is excluded (unmaintained).
- `projects/CURe/cure.py:5005-5063` validates/reuses multipass synth artifacts and persists grounding state; the post-review governor is a separate, smaller validation pass that runs after this existing flow.
- `projects/CURe/tests/test_subsequent_review.py:1-10` imports `_subsequent_review_unittest.py`; new subsequent-review suites join the public wrapper.
