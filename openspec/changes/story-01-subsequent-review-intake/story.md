# Story 01 — Subsequent Review Intake and Prior Finding Ledger

Plan: 🟢 PLAN APPROVED
Status: ✅ DONE

> Migrated from legacy `agent_coordination` into this OpenSpec change workspace on 2026-06-12.
> Runtime progress/history lives in `progress.md`; implementation review history lives in `reviews.md`.

## Purpose

Give `cure pr` the ability to collect and normalize PR discussion history plus prior CURe review findings into a single reconciled prior-finding ledger. When subsequent review is enabled (via the Control Plane), the operator sees a preflight summary of what was found; when it is disabled, legacy fresh `cure pr` behavior is preserved unchanged. Disabled or degraded modules must never produce silent suppressions or source-resolution claims.

## Actors

- Primary: CURe operator running `cure pr` on a PR that already has prior CURe reviews or PR discussion
- Secondary: implementation agent
- System: CURe subsequent-review control plane and intake modules (Control Plane, PR History Collector, Prior Review Corpus Builder, Prior Finding Extractor, Finding Reconciler)
- External service: GitHub REST/GraphQL API
- Reviewer: story-review agent

## Triggering Need

CURe already detects completed review sessions for a PR (`scan_completed_sessions_for_pr`) and has a `--if-reviewed` switch, but a fresh `cure pr` currently stages only PR metadata and does not deterministically collect PR discussion or prior review findings. The simulation fixture (`docs/examples/subsequent-review-simulation.md`) and simulated PR #21 document the desired behavior but no runtime module exists yet to ingest this data. This story establishes the foundation: the module contracts, toggles, and the prior-finding intake pipeline that stories 02 and 03 will build on.

## Expected Prerequisites

- DEPENDS = none

## Scope

- Define module contract and typed interfaces for all 13 functional modules (shared enums, dataclasses, ledger schemas, module-run status model).
- Update project packaging/type-check configuration so the new `cure_subsequent_review` package is included in the installed distribution and in the plain project `mypy` check scope.
- Implement the Control Plane: top-level subsequent-review enable/disable switch, per-module enable/disable toggles, run manifest, evidence-policy recording (`trusted`/`untrusted`), and preflight summary output.
- Preserve existing `--if-reviewed prompt|new|list|latest` branch semantics. Subsequent-review intake may run and persist artifacts only after the branch decision proceeds to a new sandbox; `list`, `latest`, and prompt-selected historical-review exits must not create new sandbox/work artifacts.
- Implement the PR History Collector: fetch GitHub issue comments, review bodies, review comments, best-effort review-thread state (`resolved`, `unresolved`, `unknown`, or unavailable marker), pagination/completeness markers, and API-failure provenance; persist as a normalized discussion artifact under `work/subsequent/`.
- Implement the Prior Review Corpus Builder: locate completed CURe sessions for the PR using existing `scan_completed_sessions_for_pr` seams, identify CURe-authored PR comments and pull review bodies when author/provenance can be determined, and build a corpus with reviewed heads, artifact/comment/review references, and provenance.
- Implement the Prior Finding Extractor: parse prior CURe review artifacts/comments into canonical prior finding candidates with IDs, severity, section, source evidence snippets, reviewed head, artifact/comment provenance, and fingerprint hints; malformed or partially parseable artifacts must record per-artifact degraded status while preserving parseable findings.
- Implement the Finding Reconciler: group repeated, superseded, and related findings; preserve canonical links and all provenance; emit an explainable reconciliation ledger.
- Convert the synthetic subsequent-review simulation into local deterministic JSON/markdown fixtures using the PR `#9999` convention. Fixture goldens cover raw prior-review A/B inputs, raw discussion C inputs, raw source-fact S inputs as fixture data, and Story 01 intake/extraction/reconciliation/degraded expectations only; source-state verification and final disposition/arbitration goldens belong to Stories 02/03.
- Disabled or degraded modules must record status and must never create suppressions or source-resolution claims.
- Persist `work/subsequent/` artifacts for new-review runs: module-run manifest, discussion artifact, prior-review corpus, extracted findings, reconciliation ledger.

## Out of Scope

- Source-verification (belongs to Story 02 — Source Truth Verifier).
- Discussion-to-finding linking, authority interpretation, and disposition actions (belongs to Story 02 — Discussion Signal Resolver, Disposition Arbiter).
- Source-state/disposition golden expectations for the simulation's S facts and final A/B/C outcomes; Story 01 may store those facts only as raw fixture inputs for later stories.
- Prompt injection and context packaging (belongs to Story 03).
- Output validation, persistence/reuse, degraded runtime UX, and landmark trace (belongs to Story 03).
- SVG polish or redesign.
- Live PR #21 integration — local fixtures only for deterministic tests.

## Scenarios / Behavior Examples

- S1: Given `cure pr` is invoked with subsequent review enabled and the run proceeds to a new sandbox, when the Control Plane preflight runs, then the operator sees a summary listing prior completed sessions, detected PR discussion activity, and module enable/disable status. Covers: A1.
- S2: Given subsequent review is disabled at the top level, when `cure pr` is invoked, then CURe proceeds with legacy fresh review behavior and no subsequent-review modules execute. Covers: A2.
- S3: Given PR History Collector is enabled, when GitHub issue comments, review bodies, review comments, and available review-thread state exist for the PR, then a normalized discussion artifact is written under `work/subsequent/` with discussion events, thread-state markers, and pagination/completeness markers. Covers: A3.
- S4: Given GitHub API access is unavailable, pagination is partial, or thread state cannot be resolved, when PR History Collector runs, then the discussion artifact records the relevant degraded status rather than silently treating missing data as empty or resolved. Covers: A4.
- S5: Given completed CURe sessions exist for the PR, when Prior Review Corpus Builder runs, then a corpus is built containing each session's reviewed head, artifact path, and provenance. Covers: A5.
- S6: Given no completed CURe sessions exist for the PR, when Prior Review Corpus Builder runs, then it records `no_prior_reviews` without failing the intake run. Covers: A6.
- S7: Given CURe-authored PR comments or pull review bodies can be identified for the PR, when Prior Review Corpus Builder runs, then those bodies are represented in the corpus with source-level provenance. Covers: A13.
- S8: Given a prior-review corpus exists, when Prior Finding Extractor runs on well-formed prior artifacts/comments, then canonical finding candidates are extracted with IDs, severity, source evidence snippets, reviewed head, artifact/comment provenance, and fingerprint hints. Covers: A7.
- S9: Given a prior-review artifact is malformed or only partially parseable, when Prior Finding Extractor runs, then it records degraded parse status for that artifact while preserving parseable findings from that artifact and other artifacts. Covers: A14.
- S10: Given prior findings include repeated or superseded items across multiple review sessions, when Finding Reconciler runs, then an explainable reconciliation ledger groups duplicates, preserves canonical links and all provenance, and records supersession claims. Covers: A8.
- S11: Given any intake module is disabled or its data source is unavailable, when a downstream module would normally consume that module's output, then the disabled/unavailable status is recorded and no suppressions or source-resolution claims are invented. Covers: A9.
- S12: Given the new `cure_subsequent_review` package exists, when the project is installed and type-checked, then the package is importable from the installed distribution and covered by `mypy`. Covers: A12.
- S13: Given completed prior review sessions exist and `--if-reviewed list` is used with subsequent review enabled, when `cure pr` runs, then CURe lists historical sessions and exits without creating a new sandbox or subsequent-review artifacts. Covers: A15.
- S14: Given completed prior review sessions exist and `--if-reviewed latest` is used with subsequent review enabled, when `cure pr` runs, then CURe prints the latest historical review and exits without creating a new sandbox or subsequent-review artifacts. Covers: A16.
- S15: Given completed prior review sessions exist and interactive `--if-reviewed prompt` is used with subsequent review enabled, when the operator selects a historical review, then CURe prints that review and exits without creating a new sandbox or subsequent-review artifacts. Covers: A17.
- S16: Given completed prior review sessions exist and `--if-reviewed new` is used with subsequent review enabled, when `cure pr` runs, then CURe proceeds to a new sandbox and runs the subsequent-review intake pipeline before the review prompt is assembled. Covers: A18.
- S17: Given completed prior review sessions exist and `--if-reviewed prompt` runs non-interactively with subsequent review enabled, when CURe falls back to a new review, then it proceeds to a new sandbox and runs the subsequent-review intake pipeline before the review prompt is assembled. Covers: A19.
- S18: Background: local fixtures derived from `docs/examples/subsequent-review-simulation.md` provide deterministic raw A/B/C/S inputs plus Story 01 extraction, reconciliation, and degraded-path goldens. Orientation only.
- S19: Given prior findings reuse local display IDs across different corpus entries and later findings cite ambiguous or chained `Supersedes:` references, when Finding Reconciler runs, then the ledger keeps local IDs namespaced by origin, preserves every origin/provenance record, records ambiguity rather than overwriting, and resolves transitive supersedes chains without splitting the full origin graph. Covers: A20.
- S20: Given `gh api --paginate --slurp` fails for an issue-comment, review, or review-comment list endpoint, when PR History Collector runs with recoverable fallback allowed, then it either retries with a compatible list collection path or records degraded status with stderr/cause taxonomy; it never reports recoverable failure as zero discussion. Covers: A21.
- S21: Given completed CURe sessions contain the actual generated human-review `review.md` format from local runs, when Prior Finding Extractor runs, then it extracts usable finding candidates from that format or consumes an explicit machine-readable finding ledger emitted by those reviews; otherwise it records parse degradation while preserving the source corpus as evidence of missing prior-finding memory. Covers: A22.

## Acceptance

- A1: When subsequent review is enabled and `cure pr` proceeds to a new sandbox, the Control Plane preflight summary lists detected prior completed sessions, PR discussion activity, and per-module enable/disable status.
- A2: When subsequent review is disabled at the top level, `cure pr` proceeds with legacy fresh review behavior; no subsequent-review modules execute.
- A3: When GitHub discussion data is available, PR History Collector writes a normalized `work/subsequent/pr_discussion.json` artifact containing issue comments, review bodies, review comments, available review-thread state markers (`resolved`, `unresolved`, or `unknown`), pagination/completeness markers, and endpoint/fetch provenance.
- A4: When GitHub API is unavailable, pagination is incomplete, `gh api --paginate --slurp` is incompatible/fails, fallback collection is incomplete, or thread state is unresolvable, PR History Collector records `discussion_unavailable`, `discussion_incomplete`, `thread_state_unavailable`, or a more specific compatible degraded status with stderr/cause detail; it never treats missing, failed, or recoverably failed data as empty or resolved.
- A5: When completed CURe sessions exist for the PR, Prior Review Corpus Builder writes a `work/subsequent/prior_review_corpus.json` artifact with each session's reviewed head, artifact path, and provenance.
- A6: When no completed CURe sessions exist, Prior Review Corpus Builder records `no_prior_reviews` status without error.
- A7: When a prior-review corpus exists, Prior Finding Extractor writes `work/subsequent/prior_findings.json` with canonical finding candidates containing IDs, severity, section, source evidence snippets, reviewed head, artifact/comment provenance, and fingerprint hints; coverage includes CURe's canonical fixture markdown and either CURe's actual generated human-review `review.md` format or a machine-readable prior-finding ledger emitted by completed reviews.
- A8: When prior findings include repeated or superseded items, Finding Reconciler writes `work/subsequent/reconciled_findings.json` grouping duplicates, preserving canonical links, every origin/provenance record, local-display-ID namespace/origin disambiguation, and supersession claims including transitive chains.
- A9: When any intake module is disabled or its upstream data is unavailable, the run manifest records the disabled/degraded status; no downstream suppressions or source-resolution claims are invented.
- A10: Module contract types (enums, dataclasses, ledger schemas, module-run status model) are defined and importable for all 13 functional modules.
- A11: Local deterministic fixture data under `tests/fixtures/subsequent_review/` covers at minimum: two prior CURe reviews with A-01 through B-06 findings, raw discussion comments C-01 through C-10, raw source-fact S-01 through S-08 fixture inputs, duplicate/superseded cases, API-unavailable path, thread-state-unavailable path, pagination-incomplete path, and parse-degraded prior artifact path. Story 01 fixture goldens cover intake/extraction/reconciliation expectations only, not Story 02/03 source-state or disposition semantics.
- A12: The new `cure_subsequent_review` package is included by project packaging and in the plain project `mypy` check scope; importability cannot rely only on running from the source checkout.
- A13: When CURe-authored PR comments or pull review bodies can be identified for the PR, Prior Review Corpus Builder represents them in `work/subsequent/prior_review_corpus.json` with comment/review ID or URL, author/provenance fields, body or artifact reference, created/submitted timestamp, review state where applicable, and reviewed head when inferable.
- A14: When prior review artifacts are malformed, partially parseable, or use an unsupported human-review format, Prior Finding Extractor records per-artifact degraded parse status with source provenance and cause, preserves parseable finding candidates, and continues processing other corpus entries without claiming full prior-review coverage.
- A15: With completed prior review sessions and `--if-reviewed list`, subsequent-review enabled mode preserves the existing list-and-exit behavior and creates no new sandbox or `work/subsequent/` artifacts.
- A16: With completed prior review sessions and `--if-reviewed latest`, subsequent-review enabled mode preserves the existing latest-review print-and-exit behavior and creates no new sandbox or `work/subsequent/` artifacts.
- A17: With completed prior review sessions and interactive `--if-reviewed prompt`, selecting a historical review preserves the existing print-and-exit behavior and creates no new sandbox or `work/subsequent/` artifacts.
- A18: With completed prior review sessions and `--if-reviewed new`, subsequent-review enabled mode proceeds to a new sandbox and runs the Control Plane/intake pipeline before review prompt assembly.
- A19: With completed prior review sessions and non-interactive `--if-reviewed prompt`, subsequent-review enabled mode preserves the existing fallback-to-new behavior and runs the Control Plane/intake pipeline before review prompt assembly.
- A20: Finding Reconciler regression fixtures cover duplicate local display IDs from multiple corpus entries, ambiguous supersedes targets, and transitive supersedes chains such as `CURE-03 -> CURE-02 -> CURE-01`; the emitted ledger preserves the complete origin/provenance graph and records ambiguity instead of overwriting or collapsing unrelated origins.
- A21: PR History Collector regression fixtures cover `gh api --paginate --slurp` failures for issue comments, reviews, and review comments; artifacts preserve stderr/exit/cause taxonomy (for example auth, transport, CLI-flag compatibility, API status, pagination shape), and documented recoverable failures use a compatible fallback or are degraded-with-cause rather than interpreted as zero discussion.
- A22: Prior Finding Extractor regression fixtures/goldens are sourced from real CURe-generated `review.md` reports from completed local runs, or from an explicit machine-readable prior-finding ledger contract emitted by CURe reviews; success requires non-empty usable findings for real generated prior reviews unless the artifact/ledger is explicitly degraded with source/cause provenance.
